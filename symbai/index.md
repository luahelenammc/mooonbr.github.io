## 1. What this file is

This file bootstraps **SymbAI Business Brain v0**.

SymbAI Business Brain is the shared AI-readable company brain for the four SymbAI Studio founders. It is meant to hold team profiles, company memory, project registry, decisions, product doctrine, business offers, dogfooding notes, governance rules, and future platform context.

It is powered by **Moon Source Business**, meaning it inherits Moon Source’s method of structured memory, routing, source governance, fact/inference separation, patch logic, export discipline, founder-sovereignty safeguards, and AI-readable continuity.

It is not Moon’s private corpus. It is not a legal company agreement. It is not an equity document. It is not a finished runtime. It is not a complete RAG/MCP/multi-agent platform yet.

Core goal:

> Build the shared business brain before building the external platform.

## 2. Core thesis

SymbAI Studio should not jump directly from technical capability into external platform-building.

The next valuable step is to create a shared internal intelligence layer that all four founders and their AIs can use.

The platform layer can come later.

Recommended order:

1. Business Brain v0
2. Shared canon and update protocol
3. Dogfooding
4. Derived examples and context compilation
5. Queryable folders / hosted context
6. Agent auth and visible permissions
7. RAG / MCP / multi-agent runtime
8. External platform / subscriptions

Operating formula:

```text
company_brain_before_platform
shared_memory_before_runtime
manual_protocol_before_agentic_automation
canon_before_generated_examples
dogfood_internal_before_external_product
moon_source_method_without_private_corpus_dump
builder_native_inside__mainstream_legible_outside
```

## 7. Context compiler doctrine

Leaf clarified that “articles” are not necessarily human-facing articles.

They may function more like:

- context files
- example files
- commentary units
- agent support documents
- RAG chunks
- compiled context material
- machine-queryable knowledge artifacts

Core idea:

> Use the strongest available model to generate rich examples and support context so weaker/cheaper agents can query better context later.

This is valid, but only with strict governance.

### Required layer separation

#### Canon

Small set of human-approved source documents, decisions, profiles, and protocols.

Authority: highest.

Purpose: source of truth.

#### Derived examples

Model-generated examples, scenarios, commentary, counterexamples, and use cases derived from canon.

Authority: secondary.

Purpose: improve retrieval coverage and agent usability.

Must be labeled:

- derived
- synthetic/example
- not primary source

#### Compiled RAG folders

Indexed/queryable bundles of canon + selected derived examples.

Authority: runtime context.

Purpose: agent query substrate.

#### Human-facing docs

Clean pages intended for founders, clients, onboarding, or public communication.

Authority: presentation layer.

Purpose: readability and communication.

Hard rule:

> Generated examples are not canon.

Safe pipeline:

```text
small canon
→ best-model derived examples
→ quality filtering
→ compiled RAG folder
→ agent queries
→ logs/evals
→ pruning and updates
```

Bad pipeline:

```text
generate hundreds of articles
→ dump into RAG
→ agent retrieves synthetic drift
→ team mistakes volume for intelligence
```

## 8. Governance before hosted context

Before using any hosted platform for the Business Brain, the team must define:

- who owns each folder
- who can read each folder
- who can edit each folder
- who can approve agent access
- what is public / internal / restricted / private
- how context is exported/backed up
- how deletion/revocation works
- how permissions are logged
- how cross-founder IP and credit are protected
- what material is forbidden from ingestion
- how Moon Source material is sanitized before entering shared SymbAI context

Minimum access levels:

- private owner only
- shared founders
- specific user/agent allowed
- public readable
- external subscriber

Minimum agent access request fields:

- requesting user
- requesting agent
- target folder
- purpose
- query scope
- duration
- read-only or write
- data retention intent

Required controls:

- approve
- deny
- approve once
- approve for session
- approve until revoked
- view request log
- revoke access
- export access history

## 9. Subscription counts as usefulness signal

Leaf added that no one knows the best design yet, and that useful design may surface naturally through subscription counts as people agree something is useful or working.

This is a valid market signal, but not the only one.

Subscription counts can indicate:

- perceived usefulness
- repeated demand
- social validation
- context value
- market interest

But subscription counts do not prove:

- correctness
- truth
- safety
- governance quality
- absence of synthetic drift
- long-term retention
- business viability by themselves

Use subscription counts as one signal among others.

Complement with:

- active usage
- repeated queries
- retention
- successful task completion
- user feedback
- low hallucination rates
- low permission friction
- low context drift
- revenue or willingness to pay
