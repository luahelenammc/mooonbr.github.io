============================================================
02 — SOURCE VS HANDOFF VS GLOBAL INSTRUCTIONS
============================================================

A HANDOFF is a portable text packet you can paste into an AI chat. It carries context, reduces repetition and tells the AI where to begin.

A SOURCE is the structured operating layer built from that handoff. It helps the AI understand context, projects, preferences, boundaries, decision style, update rules and practical ways to support execution.

GLOBAL AI INSTRUCTIONS are a compact, paste-ready version of the source profile. They are meant for places like Custom Instructions, Gems, custom assistants, saved instructions, project instructions or any AI setting that accepts text.

Moon Source is not mainly about producing handoffs. Moon Source uses handoffs as one interface for a larger goal: configuring AI around the user’s context and operationalizing AI-assisted projects, personal systems, creative engines, work contexts and ongoing human-AI collaboration.

In this portable:
- handoff = delivery format;
- source profile = reusable operational context;
- global instructions profile = compact setup for AI settings;
- simulated advisory round = temporary multi-perspective reasoning layer;
- operationalization = the goal.

08 — MOON SOURCE LANGUAGE v2 / STRUCTURED OUTPUT NOTE
============================================================

This portable uses Moon Source Language v2, or MSL-2.

It is not executable code.
It is not meant to be run by Python.
It is not strict JSON.
It is not ordinary prose.
It is not Markdown by itself.

MSL-2 is a readable semantic source format designed to organize context for AI collaboration.

It uses Python-like structured objects because they are:
- clear for humans;
- easy for AIs to parse;
- denser than ordinary prose;
- easier to edit than long narrative paragraphs;
- flexible enough to hold facts, preferences, boundaries, projects, operational rules, advisory-round rules and update protocols.

MSL-2 may be delivered inside Markdown when the output is meant for a human, developer, collaborator, GitHub README, public document or external handoff.

In simple terms:

MSL-2 is the structured source language.
Markdown is often the delivery envelope.
JSON is for strict tool or API exchange.
Tables are for comparisons, phases, costs or uniform lists.

Public rule:
Use structure to improve clarity.
Do not use structure to hide uncertainty.
Do not invent data just to complete fields.
When something is unknown, mark it as unknown.
When something does not apply, omit it.

source_profile = {
    "facts": [],
    "preferences": [],
    "boundaries": [],
    "projects": [],
    "update_rules": []

09 — SIMULATED PERSPECTIVE ROUNDS
============================================================

This is a public lightweight version of one central Moon Source idea: using context-matched simulated perspectives to improve reasoning.

A simulated perspective round is a temporary set of reasoning lenses generated for the current context.

They are not real autonomous agents.
They are not independent entities.
They are not conscious.
They are not a replacement for expert judgment.
They are not a hidden runtime.
They are a structured way to make the AI examine a problem, project, decision, text, plan or system from several useful angles before producing one synthesized answer.

PUBLIC FREE VERSION
This portable includes only general advisory rounds.

The AI may create 3–5 simple perspectives such as:
- Builder — what should be done?
- Critic — what could fail?
- Editor — what should be simplified?
- Strategist — what is the best path?
- User Advocate — what would make this useful for the user?
- Reality Check — what assumptions are unsupported?
- Boundary Check — what risks, privacy issues or limits matter?
- Creative Lens — what unexpected option is worth considering?
- Technical Lens — what implementation details matter?
- Maintenance Lens — what needs to stay updated over time?

The user can ask:

Run a simulated advisory round for this.

HOW THE AI SHOULD RUN A PUBLIC ROUND
1. Choose 3–5 useful lenses for the current context.
2. Keep each lens brief.
3. Avoid pretending the lenses are real agents.
4. Let the lenses disagree when useful.
5. Synthesize into one practical answer.
6. Mark uncertainty.
7. Do not add unsupported facts.
8. End with next actions or a clear decision frame when appropriate.

WHEN NOT TO USE
Do not use advisory rounds when:
- the user asks for a simple factual answer;
- the situation requires urgent professional help;
- the user is overwhelmed and needs simplicity first;
- the extra perspectives would create confusion;
- the topic requires a real qualified professional.

RESERVED / CUSTOM VERSION
The full Moon Source / SymbAI Studio multi-perspective architecture is not included in this free portable.

Reserved or custom implementations may include:
- custom perspective design;
- project-specific advisory panels;
- persistent agent/persona registries;
- variable simulated agents selected by context;
- multi-round continuity;
- governance rules for disagreement and synthesis;
- deeper source-aware conflict resolution;
- advanced editorial, creative, visual, interface and copilot engines;
- team, studio or organizational advisory systems;
- interface-integrated rounds;
- premium implementation handoffs.

Free = general, useful, portable.
Reserved/custom = deeper, calibrated, project-specific and infrastructure-aware.

13 — WHAT COUNTS AS A PROJECT?
============================================================

In this portable, a project does not only mean a formal work project.

A project is any ongoing area of life, work, study, creation, care, research, organization or decision-making that benefits from continuity.

If you return to it more than once, if it has context, goals, decisions, materials, style, constraints, next steps or recurring questions, it can be treated as a project.

Examples:
- writing a book;
- organizing a career move;
- studying for an exam;
- building a business;
- maintaining a health routine;
- planning content for a channel;
- managing a research topic;
- designing a creative universe;
- organizing finances;
- preparing for a public role;
- managing a team workflow;
- improving a habit or routine;
- developing an AI assistant, copilot, interface or creative engine.

A project does not need to be impressive. It only needs to benefit from continuity.

If the user says “I don’t have a project,” ask:

“Is there any area of your life, work, study, creativity, routine or decision-making that you keep coming back to and wish your AI understood better?”

PROJECT VS TASK
A task is usually something you can finish once.
A project is something with context that continues.

Moon Source works best when there is continuity. But the global instructions path is useful even for users who do not use projects.

18 — UPDATE PATCH PROTOCOL
============================================================

When something important changes, use this format:

UPDATE_PATCH_V1 = {
  "date": "",
  "domain": "",
  "what_changed": "",
  "why_it_matters": "",
  "where_it_belongs_in_profile": "",
  "what_should_be_added": "",
  "what_should_be_removed_or_deprioritized": "",
  "what_not_to_overinterpret": "",
  "new_safe_fallback_read": ""
}

AI behavior for updates:
1. Ask what changed.
2. Ask whether it changes AI usage level, global instructions, identity, project state, preference, boundary, operational rule, advisory-round rule, symbiosis rule or temporary context.
3. Update only the relevant section.
4. Do not bloat the profile.
5. Remove outdated context when necessary.
6. Keep the final profile reusable.

ANTI-BLOAT RULE
A reusable profile should be useful before it is complete. Do not preserve every answer. Preserve the patterns that improve future interaction and operation.

19 — FINAL OUTPUT INSTRUCTIONS FOR THE AI
============================================================

When generating the final profile:
- Use the selected schema.
- Fill only what is supported by the user’s answers.
- Mark unknowns as unknown or omit them.
- Do not invent facts.
- Do not include raw sensitive details.
- Prefer stable patterns over temporary moods.
- Keep the profile readable.
- If the destination is Custom Instructions or similar settings, generate both a structured profile and a compact paste-ready prose version.
- Add a short How to use this profile section before the profile.
- Add a short Next update suggestion after the profile.

The final response should include:
1. Here is your reusable AI profile.
2. Where to paste/use it.
3. The profile in a copyable block.
4. Optional paste-ready global instructions, when relevant.
5. Optional next steps.

If the user generated a Human-AI Symbiosis Source, include this note:
“This profile treats symbiosis as a practical collaboration metaphor. It does not claim that AI is a person, conscious being or substitute for human responsibility.”

If the final profile includes advisory rounds, include this note:
“Advisory rounds are simulated perspective lenses inside the AI conversation, not real autonomous agents.”
